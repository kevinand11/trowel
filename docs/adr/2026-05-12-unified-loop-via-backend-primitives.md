# Unified AFK loop driver; Backend exposes per-role primitives

> **Supersedes:** [2026-05-11-afk-loop-asymmetric-across-backends.md](./2026-05-11-afk-loop-asymmetric-across-backends.md).
>
> **Superseded by:** [2026-05-13-storage-behavior-separation.md](./2026-05-13-storage-behavior-separation.md). The per-role primitives (`prepare<Role>` / `land<Role>` / `classifySlice` / `reconcileSlices`) move off the **Backend** (renamed **Storage**) into the loop driver itself; storage becomes pure persistence. The "throw on unsupported phase" runtime invariant goes away because storages no longer expose phase methods at all — phase enablement is driven by user **Flags** gated by storage **Capabilities** (`prFlow`). The text below describes the previous design.

The **AFK loop** runs from a single driver in `src/work/loop.ts`. It iterates the actionable **Slice** queue, calls `Backend.classifySlice(slice, config)` to pick the next role for that slice, calls the backend's `prepare<Role>` method to set up the **Slice branch** and `SandboxIn`, spawns the **Sandbox** itself, then calls the backend's `land<Role>` method to apply the **Verdict**. The previously-separate `src/work/loops/file.ts` and `src/work/loops/issue.ts` files are deleted.

`Backend` gains the following required methods:

- **`classifySlice(slice, config): ResumeState`** — the loop's dispatch table. `ResumeState = 'done' | 'blocked' | 'implement' | 'review' | 'address'`. The file backend's classifier returns only `'done' | 'blocked' | 'implement'` in practice; the issue backend can return any of the five (with `'review'` and `'address'` gated by `config.work.usePrs && config.work.review`).
- **`reconcileSlices(prdId, ctx): Promise<void>`** — called once at the top of each outer iteration, before `findSlices`. The issue backend opens draft PRs for any slice with `branchAhead && !prState` (the self-heal previously surfaced as the `create-pr-then-review` resume state). The file backend implements this as a no-op — reconciliation isn't a missing capability, the backend just has nothing to reconcile.
- **`prepareImplement(slice, ctx) / landImplement(slice, verdict, ctx)`** — required on every backend.
- **`prepareReview / landReview / prepareAddress / landAddress`** — required signatures; the file backend throws `Error('not supported on file backend')`. The throws are unreachable in practice because the file backend's `classifySlice` never emits those states; the throw is a runtime invariant guarding against caller bugs.
- **`maxConcurrent: number | null`** — declarative concurrency cap. The file backend declares `1` (its implementer commits directly to the **Integration branch**, which doesn't tolerate parallel writes). The issue backend declares `null` (no backend-side cap; `config.sandbox.maxConcurrent` is the only limit). The loop applies `min(config.sandbox.maxConcurrent ?? Infinity, backend.maxConcurrent ?? Infinity)`.

`BackendDeps` is extended with a `git: GitOps` bag (`fetch`, `push`, `checkout`, `mergeNoFf`, `deleteRemoteBranch`, `createRemoteBranch`) and a `log: (msg: string) => void`. These are persistent capabilities of "this backend running against this repo" and live on the backend instance from construction, not threaded through every method call. The per-call `PhaseCtx` is `{ prdId, integrationBranch, config: { usePrs, review } }` — the values that vary per loop invocation. `findPrNumber` becomes a backend-internal helper rather than a dep; `slugify` stays a plain import.

The reversal of the prior asymmetric-loop ADR is driven by two changes that converged the loops:

1. `config.work.review` now defaults to `false` (see ADR `agent-review-opt-in`). The issue backend's primary flow is now "implement → push slice branch → open draft PR → stop," which is structurally close to the file backend's "implement → push integration branch → close slice." The four-state machine is opt-in, not the default — the asymmetry that justified two drivers narrowed substantially.
2. The interface-bloat objection from the prior ADR (file backend implementing PR-flow methods as silent no-ops, advertising a capability it doesn't have) is resolved by making the unsupported methods *throw* and by gating their reachability through `classifySlice`. The loop never invokes `prepareReview` on the file backend because `classifySlice` never returns `'review'` for a file-backend slice. Methods are required at the type level, unreachable at runtime — the interface is honest, and the loop's call sites stay free of null checks.

## Considered options

- **Keep the two-driver asymmetric design.** Rejected: with `review: false` default, the issue backend's primary flow is no longer the four-state machine. The remaining difference between the loops is ~30 lines of post-sandbox handoff per phase — duplicating outer iteration, step-cap, refetch, blocked-slice short-circuit, and concurrency batching across two drivers to preserve that ~30 lines is the wrong trade.
- **One method per backend (`processSlice(slice, ctx) → outcome`) that owns the entire per-slice progression, including the step-cap inner loop.** Rejected: the step-cap loop, refetch-between-steps, blocked-slice short-circuit, and `Promise.allSettled` concurrency batching are identical across backends. Pushing them into the backend just relocates the duplication.
- **Backend exposes `classifySlice` + `applyVerdict(role, verdict)` only; sandbox-prep stays in the loop.** Rejected: sandbox-prep is per-role asymmetric too — the issue backend's `prepareImplement` runs `gh issue develop` to create a slice branch and the addresser's prep calls `fetchPrFeedback`; the file backend's prep just returns the integration branch. Pulling prep into the loop forces backend-conditional branches there, defeating the unification.
- **Optional methods (`prepareReview?: ...`).** Rejected: callers have to null-check every invocation (`if (backend.prepareReview)`), and the type system can't distinguish "method exists but is unreachable" from "method doesn't exist." Throw-on-unsupported is honest at the type level (the method is required, so the call site is unambiguous) and the runtime invariant — "this throw is unreachable because the classifier doesn't emit this state" — is documented at the throw site.
- **Two interfaces (`BaseBackend` for required methods, `PrBackend extends BaseBackend` for PR-flow methods); loop is generic over `BaseBackend`.** Rejected: forces the loop to type-narrow before dispatching review/address, which means the loop has to know which roles need `PrBackend`. The classifier-gates-reachability invariant collapses this back to a single interface; the type split would add ceremony without buying type-level safety the runtime invariant doesn't already give.
- **Backend.maxConcurrent as a boolean `parallelizable: boolean` instead of `number | null`.** Rejected: loses information if a backend ever wants a non-1, non-Infinity cap (e.g., "I can run 3 sandboxes in parallel but not 10 because I'm rate-limited by an upstream API"). The number cap costs nothing extra and is more expressive.

## Consequences

- `src/work/loops/file.ts` and `src/work/loops/issue.ts` are deleted. Their phase logic (the `processFileSlice` / `processOnePhase` bodies, the `classifyResumeState` function) moves into `src/backends/implementations/file.ts` and `src/backends/implementations/issue.ts` as method bodies.
- The new `src/work/loop.ts` is the single driver. `src/commands/work.ts` no longer dispatches by `backend.name`; it just calls `runLoop(prdId, backend, deps)`.
- The `create-pr-then-review` state is removed from the resume-state enum; that recovery path now lives inside the issue backend's `reconcileSlices`. The classifier becomes simpler and the loop's dispatch table has one fewer case.
- Per-phase commands (`trowel implement`, `trowel review`, `trowel address`) call `backend.prepare<Role>` + `spawnSandbox` + `backend.land<Role>` directly, bypassing the classifier. On the file backend, `trowel review` and `trowel address` surface the backend's throw as a command-level error message ("review is not supported on the file backend; use `trowel work` for the implementer-only flow"). The asymmetric-loop ADR's commitment to refusal on these commands is preserved; the source of the refusal moves from a command-level `if backend.name === 'file'` to the backend method's throw.
- Future backends (e.g., a Linear-issue backend) add one file under `src/backends/implementations/`, declare which roles they support via `classifySlice`, and throw on the rest. No new file under `src/work/`. The Backend interface stays at ~15 methods rather than splitting into base/extension interfaces.
- The asymmetric-loop ADR's claim that `findSlices` is "the only generalised call" no longer holds — `classifySlice`, `reconcileSlices`, and the four `prepare<Role>` / `land<Role>` pairs are all generalised. The ADR is superseded.
