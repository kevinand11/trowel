# Sandcastle integration — public createWorktree, host-built image, mounted credentials

> **Superseded by [`2026-05-14-drop-sandcastle-for-host-exec-turns.md`](./2026-05-14-drop-sandcastle-for-host-exec-turns.md).** This ADR describes the pre-pivot wiring (sandcastle owns worktrees and container lifecycle). After 2026-05-14 trowel drops sandcastle entirely and runs each Turn as a direct `claude --print` child process inside a trowel-managed git worktree. Read this ADR for historical context only.

Trowel's **AFK loop** spawns one **Sandbox** per agent run via `@ai-hero/sandcastle`. Sandcastle owns the worktree and container lifecycles; trowel owns the image build, credentials channel, logging path, prompt rendering, IPC files, and verdict translation. The integration uses sandcastle's public `createWorktree(...) → Worktree` + `Worktree.createSandbox(...)` split-ownership pair. Sandcastle places worktrees at `<projectRoot>/.sandcastle/worktrees/<sandcastle-generated path>/` — this path is hardcoded by the library and not configurable; the original locked design (Q14) of `<projectRoot>/.trowel/worktrees/<prdId>/<sliceId>-<role>-<runId>/` is retired. Sandcastle is pinned to an exact version (not `^`); upgrades are flagged for a manual recheck since the integration depends on the shape of `Worktree`, `Sandbox`, and `SandboxRunResult`.

The per-run lifecycle:

1. Host: `sandcastle.createWorktree({ branchStrategy: { kind: 'branch', name: <branch> }, cwd: projectRoot, copyToWorktree: config.sandbox.copyToWorktree, hooks: { host: { onWorktreeReady: [] } } })` — sandcastle creates the worktree at `<projectRoot>/.sandcastle/worktrees/<...>/`.
2. Host: write `<worktree.worktreePath>/.trowel/sandbox-in.json` and `<worktree.worktreePath>/.trowel/prompt-<role>.md` (rendered host-side via `loadPrompt`, which does backend-conditional blocks sandcastle's `promptArgs` can't).
3. Host: ensure the sandbox image is built and current (`ensureSandboxImage(config.sandbox.image, ...)` — lazy `~/.trowel/Dockerfile` copy + mtime-driven `docker build`).
4. Host: `worktree.createSandbox({ sandbox: docker({ imageName: config.sandbox.image, mounts: [{ hostPath: '~/.claude', sandboxPath: '/home/agent/.claude' }] }), hooks: { sandbox: { onSandboxReady: config.sandbox.onReady.map(c => ({ command: c, timeoutMs: 600_000 })) } } })`.
5. Host: `sandbox.run({ agent: claudeCode(config.agent.model), promptFile: <abs path to prompt-<role>.md>, maxIterations: config.sandbox.iterationCaps[capKey], logging: { type: 'file', path: '<projectRoot>/.trowel/logs/<prdId>/<sliceId>-<role>-<runId>.log' } })`.
6. Host: read `<worktree.worktreePath>/.trowel/sandbox-out.json` and the `SandboxRunResult.commits.length` returned by `run()`.
7. Host: `sandbox.close()` then `worktree.close()`. Sandcastle's `Worktree.close()` preserves the worktree on disk if dirty (uncommitted changes) and removes it otherwise — useful safety net for stuck runs.

The verdict-translation table from ADR `gh-free-sandbox-host-owns-side-effects` continues to drive the post-run gh/git ops. Two refinements added by this integration:

- The `if commits` predicate in that table reads `SandboxRunResult.commits.length` from sandcastle directly rather than re-deriving via `git rev-list`. Sandcastle already computes it against `baseHead` captured at run start.
- A new coercion rule: implementer reporting `ready` with zero commits is logged and treated as `partial`. This is symmetric to the existing role-invalid-verdict coercion and catches buggy agents that claim completion without producing output.

The Docker image is built by trowel, not by sandcastle's `docker()` provider (whose `create()` calls `startContainer` directly without invoking `buildImage`). On the first `trowel work` invocation — or any subsequent invocation where the image is missing OR `~/.trowel/Dockerfile`'s mtime exceeds the image's creation time — trowel runs `docker build -t <config.sandbox.image> -f ~/.trowel/Dockerfile ~/.trowel/`. The Dockerfile itself is bundled at `assets/Dockerfile` in the trowel repo and lazy-copied to `~/.trowel/Dockerfile` on first use.

Authentication to Anthropic is via a bind-mount of the host's `~/.claude/` into the container at `/home/agent/.claude/`, read-write. The agent in the container reuses the user's existing Claude Code credentials (subscription OAuth or API key, whichever the host's `claude` CLI is set up with) and writes its session JSONL into the same host directory. No environment variable injection of secrets; no `.sandcastle/.env` written.

BACK_TO branch preservation across Ctrl-C relies on **worktree isolation**: `trowel work` never moves the main repo's HEAD during agent runs (all per-slice work happens inside dedicated worktrees with their own HEAD), so Ctrl-C is safe in the common case. The one exception is the brief host-side `gitCheckout(integration) → gitMergeNoFf → gitPush → gitCheckout(BACK_TO)` block in the `usePrs: false` flow; that block is wrapped in a tight `try/finally`. Ctrl-C inside that ~5-second window leaves the user on the integration branch and is documented as a known small gap rather than engineered around (sandcastle's docker provider registers a `process.on('SIGINT', () => { ...; process.exit(1) })` that fires before any user-space `finally` clause, making process-wide BACK_TO restore impractical without forking).

The integration imposes the following schema changes from the pre-integration TODO:

- `config.sandbox`: drop `enabled` and `dockerfile`. Final shape: `{ image, maxConcurrent, iterationCaps, onReady, copyToWorktree }`.
- `config.agent`: drop `command` and `args`. Final shape: `{ model: string }`, default `'claude-opus-4-6'` (matches sandcastle 0.5.7's `DEFAULT_MODEL` — pinning to the library default avoids drift on upgrade).
- `ensureTrowelDir` writes `worktrees/` and `logs/` into `<projectRoot>/.trowel/.gitignore`.

## Considered options

- **Use the `@internal` `createSandboxFromWorktree(...)` to keep worktrees under `<projectRoot>/.trowel/worktrees/`.** Initially locked, then retired after confirming the function is module-system-encapsulated: it lives in `dist/createSandbox.{d.ts,js}` but is not re-exported from `index.d.ts`, and sandcastle's `package.json` `exports` field allows only `.` (the curated index) and `./sandboxes/*`. TypeScript reports *"Module '@ai-hero/sandcastle' has no exported member 'createSandboxFromWorktree'."* Reaching the function would require a `pnpm patch` to add it to `index.d.ts` + the exports field — a per-version maintenance burden on top of the existing `@internal`-stability risk. Rejected.
- **Use the public `createSandbox(...)` (single-shot worktree + sandbox).** Rejected: composes the worktree and sandbox in one call, so trowel can't write `sandbox-in.json` into the worktree before the agent boots. The split-ownership flow (`createWorktree → worktree.createSandbox`) is necessary for the IPC pattern.
- **Patch sandcastle's exports to re-expose `createSandboxFromWorktree`.** Rejected: keeps the Q14 trowel-managed worktree path but adds a permanent maintenance dependency on a patched package, and the underlying API is still `@internal` (so semantics may shift even within a patched build). Cost/benefit doesn't pay.
- **Re-implement worktree-aware sandbox creation from lower primitives (`createBindMountSandboxProvider`).** Rejected: large effort to reimplement what sandcastle already does, and ongoing maintenance as sandcastle evolves. The directory naming cost of `.sandcastle/` doesn't justify the rebuild.
- **Run sandcastle in a forked child process to isolate its SIGINT handler from trowel's BACK_TO restore.** Rejected: adds a process boundary, IPC channel, and lifecycle management for a problem that worktree isolation makes mostly moot. The remaining ~5-second exposure window (host-side merge under `usePrs: false`) is documented, not engineered around.
- **Inject Anthropic credentials as `ANTHROPIC_API_KEY` / `CLAUDE_CODE_OAUTH_TOKEN` env vars via `claudeCode({ env: ... })` instead of bind-mounting `~/.claude/`.** Rejected: works, but tracks the host's `claude` CLI auth twice (once in `~/.claude/`, once in env). The bind-mount path inherits whichever auth scheme the host has set up (subscription or API key) without trowel knowing or caring which, and ensures the agent's session state lands where a host-side `claude` invocation would also write — matching the user's existing mental model.
- **Pre-render prompts inline via `sandbox.run({ prompt: <rendered string> })` instead of writing to a `prompt-<role>.md` tmpfile.** Rejected: loses the debuggable artifact in `<worktreePath>/.trowel/`. A persisted prompt file sits alongside `sandbox-in.json` / `sandbox-out.json` as part of the unified "what crossed the boundary?" view; the inline string is gone the moment `run()` returns. Cost is one extra `writeFile` per sandbox spawn — negligible.
- **Let sandcastle's `docker()` provider build the image.** Rejected because sandcastle's `docker()` doesn't build — its `create()` calls `startContainer` directly; `buildImage` is only invoked by sandcastle's CLI/init paths, not the programmatic `createSandbox` flow trowel uses. Trowel must build the image itself regardless.
- **Keep `config.sandbox.enabled: false` wired to sandcastle's `noSandbox()` provider as a debug escape hatch.** Rejected: undermines the locked "fresh Docker container per agent run … no inter-run state leakage" contract in CONTEXT.md silently. Trowel-development debugging is already served by the existing 50+ vitest tests that exercise the host-side machinery without a real sandbox.
- **Read `SandboxRunResult.commits.length` *and* re-derive via `git rev-list` for defensive cross-check.** Rejected: doubles the work without a real failure mode — sandcastle computes commits against `baseHead` snapshotted at run start, which is exactly what trowel wants. If sandcastle's internal definition of "commits" ever drifts, the verdict-translation tests catch it via the same vitest harness that already exercises the verdict table.

## Consequences

- **Public-API upgrade risk.** The integration touches `createWorktree`, `Worktree.createSandbox`, `sandbox.run`, and `SandboxRunResult` — all public. Sandcastle's semver covers these, but a personal CLI rarely re-tests after dependency bumps, so trowel pins to an exact version (`"0.5.7"`, not `"^0.5.7"`) in `package.json`. Sandcastle upgrades become flagged events: bump the pin, manually inspect the relevant `.d.ts` files, re-run the integration test, then merge.
- **Worktrees live at `<projectRoot>/.sandcastle/worktrees/`, not `.trowel/worktrees/`.** Sandcastle hardcodes the directory under `cwd/.sandcastle/`. Trowel's `ensureTrowelDir` does not gitignore `.sandcastle/` — that would require touching the project-root `.gitignore`, which the original locked design forbids (and `.trowel/.gitignore` can only ignore siblings of itself inside `.trowel/`, not parent-directory siblings). Users see `.sandcastle/` in `git status` and add it to their project-root `.gitignore` once.
- **Trowel's `addWorktree` / `removeWorktree` retire.** Sandcastle owns worktree creation and teardown. The functions and their tests are deleted as part of the spawnSandbox refactor; cleanup goes through `worktree.close()` which has useful dirty-state preservation behaviour.
- **Concurrency-safe credentials mount.** Three parallel sandboxes (default `config.sandbox.maxConcurrent: 3`) mount the same `~/.claude/` simultaneously. Claude Code writes per-session JSONL files under UUIDs in `~/.claude/projects/`, so file-level collisions are improbable. Settings reads are concurrent-safe. Cross-conversation state in `~/.claude/` is visible to every running agent — accepted because the user is delegating to their own Claude Code identity.
- **Image rebuild policy is mtime-driven.** Editing `~/.trowel/Dockerfile` triggers a rebuild on the next `trowel work`. Docker's layer cache makes the rebuild near-instant when only late layers changed. Users who want to force-rebuild can `docker rmi <config.sandbox.image>` and the missing-image branch fires.
- **`.sandcastle/.env` leak.** Sandcastle's `EnvResolver` reads `<hostRepoDir>/.sandcastle/.env` opportunistically. If a user co-locates trowel with equipped's `.sandcastle/.env` (unlikely in normal use), equipped's declared env keys merge into trowel's container. Personal CLI, single user — flagged as a non-issue but documented for future reference.
- **Ctrl-C during the host-side merge window leaves the user on the integration branch.** Documented; no engineering. A `trowel doctor` command could optionally detect and offer to switch back, but that's defer-able.
- **Logs survive in `<projectRoot>/.trowel/logs/<prdId>/<sliceId>-<role>-<runId>.log`.** No automatic rotation in v0. Users who care can `rm -rf .trowel/logs/`; future work can add a `config.work.logCleanupAge` analog to `worktreeCleanupAge`.
- **Prompt files survive until worktree removal.** `<worktreePath>/.trowel/prompt-<role>.md` is part of the worktree contents; `removeWorktree`'s `git worktree remove --force` drops it.
- **`SandboxRunResult.commits.length` becomes a load-bearing signal.** Plumbed through `spawnSandbox`'s return type alongside the verdict. The verdict-translation switch reads it for the `if commits` gates in the locked table; the new implementer coercion rule reads it too. If sandcastle ever changes how `commits` is computed (e.g. counts amend commits differently), trowel's behaviour shifts. Pinned-version policy bounds the blast radius.
- **`assets/Dockerfile` becomes a tracked, shipped artifact of trowel.** Edits to it are PR-reviewable like any other file. The repo's `.gitignore` is unchanged.
